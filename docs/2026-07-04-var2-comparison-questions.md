**作成日**: 2026-07-04
**背景**: 「市場情報を学習に使っているモデルでは回収率100%を超えており、Top-1的中率もvar1.0同様30%ほど」という
ユーザー発言を受け、`C:\Users\syugo\AI\RaceAI_var2.0.0\` を実地調査した結果をまとめる。

**2026-07-04 追記**: ユーザーから詳細な回答（コード引用付き）を受領。回答自体は誠実で根拠も明示されているが、

## 0. 結論（先に要点）

実地調査の結果、**「Top-1的中率30%」という数値の裏付けは見つからなかった**。
一方「回収率100%超え」の headline 数値（単勝ROI 151.2%・ワイドROI 162.3%）は存在するが、
**その数値の信頼性に重大な留保事項がある**（2026-07-02付けの当該プロジェクト自身の監査で、

## 1. 確認できた事実

### 1-1. 「Top-1的中率30%」に対応する数値は見当たらない

`docs/` 配下を的中率・NDCG・Spearman・rank1_accuracy等のキーワードで網羅的に検索したが、
var1.0の `evaluate.py` が出すような「予測1位が実際に1着になった割合」に相当する指標は
一切記録されていなかった。

見つかった唯一の近い指標は**単勝ベットの的中率**（＝EV/Kellyでフィルタした後のベット対象馬が
1着になった割合）で、これは「全レースの予測1位」ではなく「実際に賭けた馬」に対する的中率であり、
定義がそもそも異なる:

| 出典 | 単勝ベット的中率 | n_bets |
|------|-----------------|--------|
| `project_audit_20260617.md`（本番baseline 2025） | **11.7%** | 2,403 |
| `global_sota_comparison_20260617.md`（Fold2 2024） | 8.8% | 114 |
| `global_sota_comparison_20260617.md`（Fold3 2025） | 13.0% | 200 |

さらに `project_audit_20260617.md` には「Track B 単勝25%」という**的中率25%達成を狙った
R&Dトラックが3回連続で失敗し「構造限界」として本番非連動のR&D 20%配分に格下げされた**
という記載がある。25%到達すら「構造限界」と評価されている文脈で、30%という数値は
現在のドキュメントからは確認できなかった。

**解釈の候補**（要確認）:
- (a) ユーザーが見た「30%」はvar2.0.0の別の評価軸（例: rank系モデル単体の生の順位精度で、
  ベット戦略を通す前の指標）かもしれない。そのような指標が存在するなら教えてほしい
- (b) 単勝ではなくワイド的中率（23.4%）や複勝的中率など、券種違いの数値と混同している可能性
- (c) var1.0とは異なる評価対象期間・データセットでの数値の可能性

### 1-2. 「回収率100%超え」は事実だが、信頼性に重大な留保がある

`docs/implementation_status_20260617.md` の headline:

> 単勝: ROI 151.2% / MDD -19.9% / Sharpe 0.125 / n=2,403
> ワイド: ROI 162.3% / 的中 23.4% / n=806

これ自体はvar1.0の最良ケース（ワイドEV≥1.3で85.78%）を大きく上回る。しかし
`docs/system_audit_improvement_plan_20260702.md`（**2026-07-02付、このモデルの内部監査**）には
以下の記載がある:

1. **本番デプロイ中のbinaryモデルに、当該レース結果由来のリーク列3本が混入**していることが実測確認済み
   （`base_time_cond_zscore` / `lap_time_std` / `early_pace_ratio` — いずれも「勝者タイム」「実測ラップ」
   由来。これはvar2.0.0自身のCLAUDE.md禁止事項7「当該レースの確定結果を特徴量に使わない」に違反）
2. **main ブランチの検証システム（backtest.py / diagnostics.py / E2Eテスト）が全てimportエラーで
   起動不能**な状態が2026-06-15から続いており、「いかなる変更も合格基準で検証できない」状態
3. 本番パイプラインに障害レース除外フィルタが実質ゼロ、馬体重異常値ガードも本番になし
4. リーク除去後の再検証値（別ブランチ）は Fold2=116.0%（旧ベースライン137.0%から低下）で、
   「リーク除去の影響が有力だが原因確定は未了」と監査レポート自身が留保をつけている

つまり **151.2%/162.3% という数値は、リークが発見される前・検証システムが壊れている状態で
main ブランチが主張していた数値**であり、現時点でこの数値をそのまま「市場情報を使えば
100%を超えられる証拠」として引用するのは早計と考えられる。

### 1-3. アーキテクチャが根本的に異なる（Top-1とROIの関係が違う設計）

var2.0.0は2系統の推論を意図的に併存させている:

| 系統 | モデル | 数式 |
|------|--------|------|
| 本番（rank） | LightGBM ランキング × 3シード | rank1/2/3のソフトラベル、**init_scoreなし** |
| backtest検証（binary残差） | LightGBM 二値分類 | `raw_score + market_log_odds → sigmoid → レース内正規化` |

**本番の rank 系モデルは実は市場情報（init_score）を使っていない**。市場情報を使うのは
backtest検証専用の binary残差系のみ。つまり「市場情報を使うモデル」と一括りにする前に、
「本番運用されているrankモデル」と「バックテスト専用のbinary残差モデル」のどちらの数値を
指しているのかで、比較の意味が変わる。

さらに評価軸自体が違う。var1.0は「Top-1的中率・NDCG・Spearman」でランキング精度を測るのに対し、
var2.0.0は「ROI・MDD・Sharpe」で**ベット戦略込みの収益性**を測る。EV閾値・Kelly基準による
ベット対象の絞り込みを経ているため、「モデルの生の予測精度」と「最終的なROI」の間に
ベッティングレイヤーの効果が挟まっている（これはvar1.0のR-1検証で「Sternの改善が実は
EVフィルタの間引き効果だった」と判明したのと同種の混同ポイント）。

### 1-4. 「市場情報あり/なし」の直接比較実験が実は存在する（ただし未実行）

var2.0.0には `exp/no-market-ablation` というブランチがあり、市場列を完全除去した特徴量
（`features_past_v26_no_market.parquet`）でrank系・binary系の両方を再学習し、既存の
ensemble_v5（市場情報あり）とROI・的中率を比較する実験基盤（`compare_market_ablation.py`）が
**コードとしては実装済み**。ただし調査の範囲では**実際にバックテストを走らせた結果ファイルは
見つからなかった**（インフラのみ実装され、実行未了と見られる）。

これはユーザーの疑問（市場情報ありなしでどう違うか）に**最も直接答えられる実験**であり、
もし実行済みの結果があるなら是非見たい。

## 3. var1.0側の今後の検討への示唆

上記が確認できれば、以下の判断材料になる:

- もし「Top-1≈30%」の数値がそもそも見つからない・混同だった場合、
  「同程度の的中率でROIだけ違う」という前提自体を再検討する必要がある
- もしROI 151%/162%がリーク混入時点の数値だった場合、リーク除去後の再検証値
  （Fold2=116.0%など）が「クリーンな市場情報ありモデルの実力値」により近い可能性が高く、
  var1.0のROI最良値（ワイドEV≥1.3で85.78%）との差はまだあるが、当初思っていたほど
  絶望的な差ではないかもしれない
- no-market-ablationの結果が出れば、「market_log_odds由来の残差学習」と
  「var1.0の純粋能力スコア」のどちらがどれだけROIに寄与するかが定量的に分かり、
  R-6（var2.0.0残差学習への統合）の設計初期パラメータとして直接使える

## 4. 回答への批判的レビュー（2026-07-04）

ユーザーの回答はコード引用・ファイルパス付きで、姿勢としては誠実。ただし内容を精査すると、
「市場情報を使えば100%超え・34%達成できる」という結論を**そのまま採用するのは危険**な理由が
複数見つかった。以下、鵜呑みにできない順に記載する。

### 4-1.【最重要】34%という数値は「市場そのものの実力」と未分離

回答で提示された `top1_win_rate ≈ 34%` は、`base_margin_col = market_log_odds` を
`init_score` として**加算した後**の binary 残差モデルの出力（`model_prob`）で計算されている。
つまりこのスコアは「LightGBMの生スコア + 市場の対数オッズ」の合成値であり、
**市場オッズ単体を使うだけでも Top-1 はかなりの高さになる**ことがvar1.0自身のCLAUDE.mdに
明記されている:

> 市場ベンチマーク: 1番人気 Top-1≈30〜33%

もし var2.0.0 の binary残差モデルが「単純に1番人気を選ぶ」戦略と比べて 34% - 33% = **+1pp
程度しか上乗せできていない**なら、これは「市場情報を使えば的中率が上がる」のではなく
「市場情報がそもそも的中率33%相当の情報を持っており、モデルはほぼそれをなぞっているだけ」
という、全く違う解釈になる。

**必須の追加確認**: 同じテスト期間・同じレース集合で「単純に人気1位（オッズ最低）を選んだ場合の
Top-1的中率」を計算し、34%との差分を出すこと。この差分こそが「モデルが市場に対して持つ
真の付加価値」であり、34%という数値単体には意味がない。差分が小さければ（例えば1〜2pp）、
「市場情報モデルがvar1.0を圧倒している」という前提そのものが崩れる。

### 4-2.【重要】var2.0.0 自身の合否基準に照らすと F3 は「リジェクト」

回答の表:

| Fold | ROI | top1_win_rate | n_bets |
|------|-----|----------------|--------|
| F1 | 186.4% | 33.9% | 153 |
| F2 | 136.8% | 34.9% | 96 |
| F3 | **98.2%** | 33.4% | 147 |

var2.0.0 自身の CLAUDE.md（このドキュメントの調査時に読んだ内容）:

> 回収率（ROI）: 合格 105%以上 / 要改善 100〜105% / **リジェクト 100%未満**
> 禁止事項4: **回収率100%未満のモデルをリリースしない**

F3（2025年〜2026年、var1.0のテスト期間と最も重なる部分）は **98.2% で var2.0.0 自身の基準で
「リジェクト」**。つまり「市場情報モデルは100%を超えている」という当初のユーザー発言は、
**3フォールド中もっとも新しい・var1.0と期間が近いフォールドに限れば成立していない**。
さらに F2 は n_bets=96 で、var2.0.0 自身の禁止事項5（200件未満は有意主張禁止）に抵触する
サンプルサイズであり、136.8%という数値も本来「判定保留」に分類されるべきもの。

3フォールドのうち胸を張って「合格」と言えるのはF1（186.4%、n=153で200件基準もギリギリ未達）
だけであり、これも複数のwalk-foldの中の1つを恣意的に取り出しているだけの可能性がある。

### 4-3. headline（151.2%/162.3%）は Optuna 探索を経た数値 — 最適化リークの懸念

回答は「151.2%はrank本番評価パイプラインのheadline」で「backtest.pyのwalk-forwardとは別系統」
と説明している。しかしこの rank 本番モデルは var2.0.0 自身の記述によれば
`n_trials=50`（Optunaハイパーパラメータ探索）を使っており、しかもvar2.0.0のCLAUDE.md自身が

> Optuna はバリデーション期間に過学習するため、binary残差モデルの学習では
> `backtest_n_trials=0` を維持する

と、**Optuna探索そのものがバリデーション期間への過学習源になりうる**ことを認めている
（だからこそ binary 系は n_trials=0 に固定している）。一方 headline の rank モデルは
n_trials=50 を使っており、この「2025 OOF」がOptunaの探索対象期間と重なっていないか、
重なっているならこの151.2%はハイパーパラメータレベルでのリークを含んでいる可能性がある。
この点は回答で解消されておらず、追加確認が必要。

### 4-4. 数値が報告のたびに大きく変動している（パイプライン不安定の兆候）

同じ「walk-forward F2/F3」に相当する数値が、参照時点によって以下のように変わっている:

| 参照時点 | F2 ROI | F3 ROI |
|---------|--------|--------|
| 2026-06-17（監査前 baseline記載） | 137.0% | 149.4% |
| 2026-07-02（監査、fixブランチ再検証） | 116.0% | 152.1% |
| 2026-07-04（今回の回答、backtest_results_20260703.json） | 136.8% | **98.2%** |

回答は「監査記載のF2=116.0%/F3=152.1%はfixブランチ時代のスナップショットで現行アーキテクチャ
では再現対象外」と説明しているが、これは裏を返せば**「特徴量・モデルが変わるたびに数値が
大きく動く」ことを認めている**ことになる。特にF3は152.1%→98.2%と50pp以上動いている。
var1.0側でも「モデルバージョンが変わるたびにROIが数pp単位で動く」ことは経験済み（A-1/A-2実験
参照）だが、50pp規模の変動は「たまたま条件を変えたら良い数値が出た／悪い数値に戻った」という
再現性の低さを強く示唆する。単発の良い数値（この場合151.2%やF1=186.4%）を代表値として
引用するのは、var1.0が雨条件で一度犯した過ちと同じ構造のリスクがある。

### 4-5. 「本日 backtest.py が動いた」の再現条件が不明

回答は「本日mainブランチでbacktest.pyを実行しF1完走を確認した」としているが、2026-07-02監査は
「mainはimportエラーで起動不能（2026-06-15から継続）」と明記していた。以下のいずれかのはずである:
(a) 監査後に誰かが黙って修正した、(b) 実行したのは実は`main`ではなく別ブランチ／ローカルの
未コミット状態だった、(c) 監査の記述自体が不正確だった。**どの分岐なのか確認しないと、
「今動いているbacktest.pyの数値」がどのコミット・どの特徴量ファイルに対応するものか特定できない**。

### 4-6. Top-1計算の母集団が var1.0 と揃っているか未確認

`calculate_ranking_metrics()` の `top1_win_rate` が、オッズ欠損・スクラッチ馬を含む
**全レース**で計算されているのか、それとも`model_prob`（market_log_odds依存）が
計算できないレースを暗黙に除外しているのか、回答からは読み取れない。var1.0のTop-1は
grade_code/abnormal_code/頭数フィルタのみで、オッズ有無によるレース除外は一切行っていない
（そもそもオッズを見ていないため）。もしvar2.0.0側がオッズ欠損レースを分母から除外していれば、
その分だけTop-1が下駄を履いている可能性がある。

### 4-7. 評価軸としての皮肉な一致

回答が認めている「151.2%はcalibrator（isotonic）+ EV/Kelly戦略込みの合成効果」という点は、
皮肉にもvar1.0がR-1検証で得た教訓（"Sternの改善に見えたものは実はEVフィルタの間引き効果だった"）
や、feedback_ev_strategy.mdに記録済みの教訓（"キャリブレーション手法は2024年にフィットしても
2025+に転移しない"）と同じ構造のリスクを抱えている。var2.0.0側でもキャリブレーション手法の
年次転移が保証されていない以上、151.2%という一時点の成功が今後も続く保証はない。

### まとめ: 現時点で言えること

- 「市場情報を使えば的中率30%・回収率100%超えを達成できる」という主張は、**部分的には裏付けが
  あるが、そのまま採用するには早い**。特に (a) 34%が市場の実力とどれだけ差があるか未検証、
  (b) 直近フォールド（F3）は var2.0.0 自身の基準でリジェクト、(c) headline数値は最適化リークの
  懸念が未解消、(d) 数値自体が測定のたびに大きく変動、という4点がクリアされない限り
  「参考にすべき成功例」として扱うのはリスクがある
