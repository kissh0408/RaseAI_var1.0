## 2. 詳細と修正方法

### A-1. `hist_sire_dist_diff` の全期間統計リーク（最優先）

**場所**: `pure_rank/src/create_features.py:434-447`（`_build_sire_features` 内）

```python
# ── 父産駒の平均勝ち距離との差（全期間統計、距離適性は安定のため許容） ─────
sire_wins = df[df["is_win"] == 1]          # ← 全期間（テスト期間含む）の勝利
sire_avg_dist = sire_wins.groupby("sire_id")["distance"].mean()
df["hist_sire_dist_diff"] = (df["distance"] - df["_sire_avg_win_dist"]).abs()
```

**問題点**:
- `df` はテスト期間（2025〜2026）を含む全データ。**2026年の勝利距離が2020年のレースの特徴量に混入**している。
- プロジェクト憲法 Rule 2（時系列リーク防止）・Rule 3（後出しじゃんけん禁止）の明確な違反。
- メモリ記録上、v3（28.9%）が「旧式全期間血統特徴量を含む非準拠モデル」と認定された経緯があるが、
  `hist_sire_dist_diff` だけが現行コードに残存している（v36_course manifest でも列存在を確認済み）。
- コメントの「安定のため許容」は憲法上の例外規定ではない。effect size は小さい可能性が高いが、
  「Top-1>30% 達成」の主張の信頼性を毀損するため除去必須。

**修正方法**（他の `_ts` 特徴量と同じ「日次集計 → cumsum → shift(1)」パターン）:

```python
# 父産駒の勝ち距離の時系列累積平均（当日を除外）
win_dist_daily = (
    df[df["is_win"] == 1]
    .groupby(["sire_id", "race_date"], observed=True)
    .agg(d_dist_sum=("distance", "sum"), d_wins=("distance", "count"))
    .reset_index()
    .sort_values(["sire_id", "race_date"])
)
grp_w = win_dist_daily.groupby("sire_id", observed=True)
win_dist_daily["cum_dist"] = grp_w["d_dist_sum"].cumsum()
win_dist_daily["cum_wins"] = grp_w["d_wins"].cumsum()
win_dist_daily["avg_win_dist_prev"] = (
    grp_w["cum_dist"].shift(1) / grp_w["cum_wins"].shift(1)
)
# ※ 勝利日ベースの sparse な系列なので、メイン df へは merge_asof(backward) で
#    「当該レース日より前の最新値」を引き当てる（by="sire_id"）
```

マージ後 `hist_sire_dist_diff_ts = (distance - avg_win_dist_prev).abs()` とし、旧列は削除する。

**検証手順**:
1. 修正後に再生成 → v37 系として保存（v33 はバックアップ保持）
2. 再学習 → Top-1 が 30.37% から大きく低下しないか確認（低下しても憲法遵守を優先）
3. `is_win` と新特徴量の当該レース相関がゼロ近傍であることを確認

### A-2. `hist_best_time_same_cond` の距離カテゴリ混在バグ

**場所**: `create_features.py:182-189`

```python
df["hist_best_time_same_cond"] = (
    df.groupby(["ketto_num", "distance_category", "surface_code", "track_condition_code"])
    ["racetime"].transform(lambda x: x.shift(1).expanding().min())
)
```

**問題点**:
- 2026-06-30 に修正した「速度指数バグ」（`distance_category` で集計 → 疑似信号化、
  finish_rank との相関 r=-0.038 とほぼ無意味）と**同じ構造**が、この特徴量には残っている。
- 同一カテゴリ内でも 1000m と 1400m では走破タイムが 20 秒以上違う。
  「過去最速タイム」の生値は**能力ではなくどの距離を走ったか**を反映する。
- LightGBM が distance と組み合わせて部分的に補正できるため致命傷ではないが、
  ノイズ源であり、速度指数修正時の知見（相関 10 倍差）から見て情報量はほぼゼロの疑い。

**修正方法**（いずれか）:
1. **推奨**: groupby キーを `distance_category` → `distance`（実距離）に変更（速度指数修正と同じ）。
   ただし `hist_speed_idx_cond_best`（実距離×馬場の標準化済み最高値）と重複するため、
   変更後に相関を確認し **r > 0.8 なら列ごと削除**が妥当。
2. 削除のみ（`hist_speed_idx_cond_best` が上位互換のため）。

**検証手順**: 修正前後で `hist_best_time_same_cond` vs `finish_rank` の Spearman を比較。
現状ほぼゼロなら削除、修正で有意になるなら保持を判断。1 変更 1 実験の原則を守る。

### C-2. 予測スコア同点時の Top-1 判定

`sort_values("pred_score", ascending=False)` は同点時に元の行順序（=馬番順）で先頭を決める。
15 モデル平均の float なので同点はまれだが、決定性を保証するなら
`sort_values(["pred_score", "horse_num"], ascending=[False, True])` のように tie-break を明示する。
`idxmax()`（supplementary 側）も「最初の最大値」を返すため同じ性質。**両者の tie-break を揃える**こと。

補足: 評価用 NDCG は `2^label - 1` のゲインで、学習側 label_gain [0,1,3,7,15,31,100] と定義が異なる。
これは「固定の外部指標」として一貫していれば問題ないが、既知の仕様として明記しておく
（学習中の valid ndcg@3 と evaluate.py の NDCG@3 は直接比較できない）。

### D-1. エラー分析が示す弱点条件（次の精度向上の主戦場）

`error_analysis_v33.json`（テスト 4,775 レース）より、全体 30.37% に対して:

| 条件 | Top-1 | 差分 | n |
|------|-------|------|---|
| **course_code=3（福島）** | **22.4%** | **-7.9pp** | 312 |
| **course_code=10（小倉）** | **23.2%** | **-7.1pp** | 383 |
| 17頭以上 | 25.7% | -4.7pp | 421 |
| 短距離（〜1400m） | 27.7% | -2.7pp | 1,675 |
| オープン（grade=5） | 27.8% | -2.6pp | 1,101 |
| 重賞（grade=7） | 27.6% | -2.8pp | 98（小サンプル） |
| 5-8頭 | 41.5% | +11.1pp | 277 |

**読み取り**:
- 最大の弱点は**ローカル小回りコース（福島・小倉）**。両場は開催期間が短く、
  馬・騎手のコース実績サンプルが少ないため `hist_same_course_win_rate` /
  `hist_jockey_course_win_rate` の NaN 率が高い（≒コース適性情報が効かない）と推定される。
- 17頭以上・短距離はペース・展開・枠の影響が大きく、能力序列だけでは当たりにくい構造。
- 馬場状態（不良 +4.2pp / 稍重 +3.0pp）は**弱点ではない**。

**改善策（優先度順）**:
1. **コース形態メタ特徴量**: course_code ごとの「直線長・回り・坂」等の静的属性、または
   「小回り(福島/小倉/中山内回り等)フラグ × hist_front_running_pref」交互作用。
   小回りは先行有利のバイアスが強く、市場が知る情報だがモデルには未提供。
2. **枠×コースの精緻化**: 現行 `wakuban_surface` は芝/ダートの符号だけ。
   短距離・多頭数で効く「コース別の枠バイアス」を、学習期間データの枠別勝率
   （course×distance_category×wakuban の時系列集計）として追加。
3. **多頭数対応**: `relative_post_position` はあるが、17頭+ では
   「先行馬密度 × 脚質」の交互作用が効く可能性。field_front_runner_density は既存なので
   `hist_front_running_pref × field_front_runner_density`（テン争い激化ペナルティ）を追加候補に。

いずれも学習期間データのみで設計判断すること（テストの弱点分析は「どの軸を探すか」の指針に留める。
閾値・条件の後付け最適化は Rule 3 違反）。

### E-1. EV/ROI 戦略の構造的限界（記録済み知見の確認）

これまでの試行（キャリブレーション 3 手法・市場乖離ベット・EV サンプル重み・EV-max 選択）は
すべて不採用。根本原因は一貫している:

```
モデル p_wide ≈ 市場が織り込み済みの情報 → EV ≈ 0.98 前後に収束 → フィルタしても ROI 改善せず
```

- Harville 確率はロングショットの P_wide を系統的に過大評価（最大 16.6pp）
- 2024 でフィットしたキャリブレーションは 2025+ に転移しない（分布シフト）

**推奨方針**（新規実験の前に必読）:
1. ROI 100% 超えを EV フィルター単独で狙う実験は**凍結**する（同型の失敗を繰り返さない）
2. ~~唯一の生存候補 weather_code=3（雨）× EV>=1.0: ROI 126.8% は n=60 で判断保留。ベットせずサンプルを蓄積し、n>=100 到達時点で bootstrap CI を再計算する~~
   **【2026-07-03 更新・クローズ】** `docs/specs/2026-07-04-roi-improvement-roadmap.md` 1.2節で、
   weather_code=3 の公式 VALID(2024) スクリーニング結果（`ev_lift=-17.1pp`、有効条件リスト不算入）が確認された。

## 5. 回収率 100% 超え問題の修正方法（E-1 深掘り・2026-07-02 追記）

> **前提の確認**: プロジェクト憲法上、ROI は合否判定軸ではない（禁止事項 3）。
> 本節の施策はすべて**ベッティングレイヤー（simulate_ev.py / predict.py）側の改善**であり、
> ランキングモデルの合否は従来どおり Top-N/NDCG/Spearman で判定する。
> オッズの参照はベッティングレイヤーでのみ許可（特徴量への混入は引き続き絶対禁止）。

### 5.1 なぜ 100% を超えられないのか — 定量分解

```
ワイドの控除率 ≈ 22.5%  →  無作為ベットの期待 ROI ≈ 77.5%
現状の最良（EV>=1.0 フィルタ）      ROI = 85.7%
                                    ─────────────
モデルが市場に対して持つエッジ      ≈ +8.2pp
100% までに必要な追加エッジ        ≈ +14.3pp（現エッジの約2.7倍）
```

つまり「モデルに優位性がない」のではなく、**優位性(+8pp)が控除率(22.5%)より小さい**のが本質。
修正の方向は論理的に3つしかない:

| 方向 | 内容 | 対応施策 |
|------|------|---------|
| (i) 確率推定の質を上げる | Harville のロングショット過大評価（最大16.6pp）を構造的に直す | R-1 |
| (ii) ハードルを下げる | 控除率の低い券種・市場の歪みが大きい局面に賭ける | R-2, R-3 |
| (iii) 市場にない情報を足す | 市場が織り込みにくい情報源を増やす／市場情報と統合する | R-4, R-6 |

これまでの失敗（キャリブレーション3手法・divergence argmax・EV-max・EVサンプル重み）はすべて
「(i) を年次依存の後付け写像でやろうとした」か「(ii) を argmax でやろうとした」もの。
以下の施策は失敗の根本原因（年次分布シフト・高オッズバイアス）を回避する設計にしている。

---

### R-1. Harville 公式の置き換え — Stern / べき乗割引モデル（優先度: 最高）

**現状の問題**: `predict.py` の Harville 公式は「1着確率の比がそのまま2着以降にも保存される」
と仮定するが、実際は下位着順ほど番狂わせが増えるため、**穴馬の連対・複勝確率を系統的に過大評価**する
（ev_diagnosis_v2: 全 bin で predicted > actual、最大 16.6pp）。EV-max が的中率 1.1% に崩壊した
直接原因でもある。

**なぜキャリブレーションと違って有望か**: Platt/Isotonic は「2024年の誤差曲線」を丸ごと写像する
ため分布シフトで死んだ。Stern 型は**構造パラメータ1個**（着順が下がるごとの割引率 λ）だけを推定
するため過学習の余地が小さく、年次転移に強い。

**実装方法**（predict.py の Harville 部分を置換）:

```python
# Harville:  P(j 2nd | i 1st) = p_j / (1 - p_i)
# Stern型:   P(j 2nd | i 1st) = p_j^λ / Σ_{k≠i} p_k^λ   （0 < λ <= 1）
# λ=1 で Harville に一致。λ<1 が「2着争いは1着争いより番狂わせが多い」を表現する。
def stern_second_prob(p: np.ndarray, winner_idx: int, lam: float) -> np.ndarray:
    q = p ** lam
    q[winner_idx] = 0.0
    return q / q.sum()
```

- **λ の推定はテスト期間を使わない**: 学習期間（〜2023）+ バリデーション（2024）の実着順に対する
  2着・3着の log-likelihood 最大化で λ2（2着用）・λ3（3着用）を1回だけフィットし、
  `train_config.json` の `plackett_luce` セクションに保存する（T_opt と同じ管理方式）
- **検証**: 2024 でフィット → 2025+ テストで (a) P_wide のキャリブレーション誤差が 16.6pp から
  縮小するか、(b) EV>=1.0 フィルタの ROI が 85.7% から改善するか、の2点で判定
- **失敗基準**: λ ≈ 1.0 に収束する（=Harville で十分だった）、または誤差が縮んでも ROI が
  改善しない場合は「キャリブレーション精度≠EV有効性」の既知パターンとして撤退

### R-2. 券種の見直し — 複勝市場への展開（優先度: 高）

**着眼点**: 現在の EV 検証はワイド・馬連（控除率 22.5%）に集中しているが、
**単勝・複勝の控除率は 20%** でハードルが 2.5pp 低い。さらにモデルの実測値から:

```
予測1位馬が3着以内に入る率（top3_coverage_rate） = 62.5%
複勝 ROI = 0.625 × 平均複勝オッズ
  → 平均 1.60 倍で ROI 100%（複勝 1.6 倍は「やや人気薄寄りの中位人気」水準）
```

つまり**複勝は、現行モデルの実力で損益分岐点に最も近い市場**である。
的中率 62.5% は分散も小さく、条件フィルタの統計的検証（n の蓄積）も速い。

参考: 単勝は Top-1=30.4% × 市場中央値 2.25 倍 ≈ ROI 68% で複勝より遠い
（モデルの選ぶ馬は人気馬に寄るため単勝配当が低い）。**優先すべきは複勝 > ワイド > 単勝**。

**実装方法**:
1. **データ前提**: JV-Link **O1 レコード（単勝・複勝の事前オッズ）**の取得スクリプトを
   `common/data/src/` に追加する（WideOdds/QuinellaOdds CSV と同じパターン。
   これが未取得なのが現在このパスを検証できない唯一の理由）
2. `simulate_ev.py` に複勝 EV を追加: `EV_place = p_place × place_odds`
   （p_place は predict.py の Harville/Stern 3着以内確率をそのまま利用）
3. 分析設計: 複勝オッズ帯別（例: 〜1.3 / 1.3〜1.6 / 1.6〜2.5 / 2.5〜）に
   的中率と ROI を集計し、「p_place が高いのに複勝オッズが 1.6 倍以上残っている」
   ゾーンの存在を確認する
4. **閾値の決定は 2024 バリデーションのみ**で行い、2025+ は前向き検証（Rule 3 遵守）

### R-3. オッズ帯×乖離閾値フィルタ — divergence 失敗の正しいリベンジ（優先度: 中）

**過去の失敗との差分**: market divergence は「argmax で1点選ぶ」設計だったため
高オッズバイアスを除去できず ROI 50.4% で死んだ。メモリの教訓どおり
**「argmax ではなく閾値フィルタ」+「オッズ帯内で比較」**に設計を変える。

**実装方法**:
1. ベット候補は従来どおり **P_wide-max ペア**（的中率 29% を維持できる唯一の選択規則）に固定
2. その候補に対し、オッズ帯 b ごとに定義した乖離条件でベット可否だけを判定する:

```python
# bracket_boundaries = [3.0, 8.0, 20.0]（config に既存）を流用
edge = p_model_wide * prior_odds_wide          # = EV
bet = edge >= (1.0 + delta[band(prior_odds_wide)])
```

3. `delta[band]` は **2024 バリデーションで各帯の実測 ROI が 100% を超える最小マージン**として決定し、
   config に固定 → 2025+ で前向き検証
4. 期待される効果: 低オッズ帯（市場の本命側、favorite-longshot bias で過小評価されがち）に
   ベットが集中し、高オッズ帯の「Harville 過大評価×高配当」の罠を帯別 δ が防ぐ
5. **R-1（Stern 化）の後に実施**すること。確率側の系統誤差を直してからフィルタを設計しないと、
   δ が Harville の歪み補正を兼ねてしまい年次転移で再び壊れる

### R-4. 市場が織り込みにくい情報の追加 — 当日馬場バイアス・展開（優先度: 中）

**理屈**: 市場（オッズ）は「能力+血統+騎手+調教+パドック+関係者情報」の集合知であり、
純粋能力特徴量の改善はほぼ織り込み済み。しかし**当日のトラックバイアス**
（今日の馬場は内が伸びる/先行有利か）は、市場でも織り込みが遅れることが知られている数少ない領域。

**実装方法**（特徴量なので通常の planner→implementer→evaluator フローで実施）:
1. `same_day_front_bias`: 同日・同競馬場・同馬場種別の**先に行われたレース**
   （race_num が小さいレース）における「4角先頭馬の平均着順」等。
   レース発走前に観測可能な情報であり、オッズではないため憲法違反ではない
2. リーク設計が繊細な点に注意: **当日の後のレース・当該レース自身を絶対に含めない**
   （race_num でソートし、同日内は `shift` ではなく「race_num より小さい行のみ」の集計にする。
   第1レースは NaN で正しい）
3. 展開系: `hist_front_running_pref × field_front_runner_density`（テン争い激化の交互作用）は
   D-1 の多頭数弱点対策と共通なので、そちらの実験に相乗りさせず**別実験として**順次検証
4. 合否は従来どおり Top-1 ゲート。副次的に ROI が動くかを simulate_ev で観測する

### R-5. 条件フィルタの前向き検証プロトコル（優先度: 中・コスト極小）

weather_code=3（雨）× EV>=1.0 の ROI 126.8%（n=60）のような「テストで見つけた好条件」は
後付け選択バイアスの疑いを排除できない。恒久ルールとして以下を定める:

1. **条件の事前登録**: 検証したい条件（例: 雨・特定オッズ帯・特定コース）は
   2024 バリデーションの分析だけを根拠に `docs/specs/` に登録してから、テスト期間で計測する
2. **判定基準の固定**: n >= 100 かつ bootstrap 95% CI 下限 > 100% で初めて「採用候補」
3. **蓄積の自動化**: 週次で新レースの EV 結果を追記する集計スクリプトを用意し、
   登録済み条件の n と CI を機械的に更新する（判断の入り込む余地をなくす）

### R-6. RaceAI_var2.0.0 残差学習との統合 — 回収率目的の本命（優先度: 高・工数大）

**位置づけ**: CLAUDE.md 末尾に「Top-1>30% を達成した場合、var2.0.0 の残差学習と組み合わせた
最終系への発展を検討する」と明記されており、**その発動条件（30.37%）は既に満たしている**。

**なぜこれが ROI の本命か**: 100% 超えに必要なのは「市場より正確な確率」であり、
市場確率を無視したままでは市場の持つ情報（パドック・直前情報・集合知）をすべて捨てて戦うことになる。
一方 var1.0 の純粋能力スコアは**市場と独立に作られた信号**なので、
「市場確率 + 純粋能力スコアの残差補正」という形で組み合わせたときに初めて価値が最大化する:

```
最終確率 = f(市場オッズ由来の確率, var1.0 純粋能力スコア)
           ── var2.0.0 側（init_score + binary residual）で学習
```

**実装方法**:
1. var1.0 側の成果物として「全レース×全馬の ensemble スコア（リーク無し・市場情報無し）」を
   parquet でエクスポートするスクリプトを追加する（`pure_rank/src/export_scores.py`）
2. var2.0.0 側で、market_log_odds を init_score にした binary モデルの**特徴量に
   var1.0 スコアを追加**して残差学習する（var1.0 のリポジトリ・憲法は一切変更しない。
   市場情報を扱うのはあくまで var2.0.0 側）
3. 評価は var2.0.0 側の軸（ROI）で行う。var1.0 の Top-1 追求はこれと並行して継続できる
4. **境界の明確化**: var1.0 に市場情報が逆流しないよう、統合作業は var2.0.0 のリポジトリでのみ行う

---

### R-7. セグメント別「対市場」診断 — 撤退基準を作る（優先度: 高・最初にやる）

R-1〜R-3 に着手する前に、**そもそも市場に局所的にでも勝てているのか**を1回だけ測定する。

**実装方法**（`pure_rank/src/diagnose_vs_market.py` 新規・ベッティングレイヤー）:
1. 2024 バリデーションで、モデル勝率 p_model と市場インプライド確率
   `p_market = (1/win_odds) / Σ(1/win_odds)` の **log-loss / Brier を比較**する
   （O1 事前オッズ取得後。R-2 のデータ前提と共通）
2. 全体だけでなくセグメント別（オッズ帯×コース×距離×頭数）に比較し、
   「p_model が p_market に勝つセグメント」を列挙する
3. **意思決定ルール**:
   - 勝てるセグメントが存在する → そこに R-3 のフィルタを絞る（根拠のある条件登録）
   - **どのセグメントでも勝てない → EV フィルタ経路（R-3, R-5）は打ち切り、
     R-6（var2 統合）に全振りする**

この診断は「EV フィルターで ROI を改善できるか」という2年来の問いに白黒をつける停止基準であり、
同型の失敗実験を今後繰り返さないための最重要ステップ。
