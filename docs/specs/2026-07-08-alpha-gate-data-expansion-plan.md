## Track A — α ゲート評価基盤（最優先。B/C 両方の前提インフラ）

### A-1 目的

「候補特徴量（またはスコア）が**市場条件付きで尤度を足すか**」を単一コマンドで判定する
標準ハーネスを作る。以後の L1 実験はすべてこれを通す。

### A-2 実装: `evaluation/alpha_gate.py`

```
入力:  candidate parquet（race_id, horse_num, cand_score）
       + fold2 OOS スコア（既存 scores_v39_course_slim_fold2_oos.parquet）
       + 単勝オッズ（evaluation/odds_loader）
モデル: p_i ∝ exp(α·z_i + β·ln q_i + γ·x_i)
       x = 候補スコア（レース内 z 標準化）。既存 fit_fusion の3パラメータ拡張
判定:  fit = 2023-01-01〜2024-12-31（oos_protocol.FIT_START/FIT_END を再利用）
       γ の尤度比検定（H0: γ=0）p < 0.01 かつ ΔLL/race を報告
出力:  evaluation/reports/alpha_gate_{candidate}.json
```

- 既存 `prob_fusion/src/fit_fusion.py` の `fusion_probs`/`fit_fusion_mle` を3パラメータに
  一般化する（`alpha` をベクトル化するのが最小変更。TDD: 2→3パラメータの回帰テスト必須）
- **TEST(2025+) は使わない。** ゲート判定は fit 期間内の交差検証（2023 fit → 2024 eval）で行う。
  TEST は「複数候補が γ 有意になった後の最終確認」で各候補 1 回のみ

### A-3 L1 実験プロトコルの改定（CLAUDE.md 更新指示）

| 段階 | 従来 | 新プロトコル |
|------|------|-------------|
| 一次審査 | Top-1 が v39 比 +0.1pp 以上 | **α ゲート: γ の LRT p<0.01（2024 eval）** |
| 二次審査 | — | Top-1 ≥ 29.99%（fold2 基準の退行防止。向上は不要） |
| 相関ゲート・リークゲート | 継続 | 継続（Top-1>40% 停止、\|r\|≥0.7 停止） |

- CLAUDE.md の「評価基準」節に本表を追記し、「Top-1 合格基準」は退行防止条件に格下げする
- 禁止事項に追加: 「**α ゲートを通らない特徴量を Top-1 改善のみを根拠に採用する**」

### A-4 合格ゲート（Track A 自体の完了条件）

- 3パラメータ MLE の単体テスト（合成データで γ の符号・0 復元）パス
- 既知のネガティブコントロール（乱数列 x）で γ 非有意、
  ポジティブコントロール（`finish_rank` から作った疑似リーク列）で γ 有意 & リーク警告が出ること

## Track B — 調教データ深掘り（HC/WC 生時系列）

### B-1 仮説（なぜ市場が知らない可能性があるか）

市場は調教の「印・評価・パドック」を織り込むが、**個体内の時系列変化**
（強度トレンド・頻度変化・加速度プロファイルの変化）は集約された形でしか
流通していない。v50 の失敗は「水準の正規化」（基準時計差）が既存 trn_* と
重複したため。**「水準」ではなく「変化・頻度・形状」の軸に限定する。**

### B-2 候補ファミリ（事前登録制。1 本ずつ α ゲート審査）

| ID | 特徴量 | 設計 | 却下済み類型との差分 |
|----|--------|------|--------------------|
| B-1 | 調教強度トレンド | 直近30日 `hc_4f_sec` の個体内回帰傾き（速くなっている馬） | v50 は水準差。傾きは未検証 |
| B-2 | 調教頻度の個体内変化 | 前走からの本数 ÷ 同馬の過去中央値本数 | 頻度軸は完全未検証 |
| B-3 | 加速度プロファイル変化 | `hc_accel_sec` の直近3本平均 − 個体キャリア平均 | v50 の accel は「ベスト水準」のみ |
| B-4 | ラスト1F失速率 | `hc_200_sec` / (`hc_3f_sec`/3) の直近トレンド | 未検証 |
| B-5 | WC×HC 併用切替 | 調教場・コース使用パターンの変化（WC は 2021+ サブセット評価） | 未検証 |

### B-3 実装規約

- リーク防止: `merge_asof(allow_exact_matches=False)`（training_date < race_date）を継続
- 各候補は `docs/specs/` に 10 行程度の事前登録（仮説・計算式・却下条件）を書いてから実装
- 審査: A-3 新プロトコル（α ゲート一次 → Top-1 退行防止二次）
- **Top-1 が上がっても γ 非有意なら不採用**。逆に Top-1 不変でも γ 有意なら採用し、
  L2 の統合変数（γ 項）として本番化する（L1 モデル再学習は必須ではない点に注意 —
  αゲートの γ 項はそのまま L2 の追加信号スロットになる）

### B-4 撤退基準

事前登録した候補 5 ファミリすべてが γ 非有意 → 「JV-Link 調教データからの市場超え信号は
現行粒度では不成立」と記録し、Track C の結果と合わせて §5 の全体判断へ。

## Track C — 複勝市場の測定（最速・数日で価値検証可能）

### C-1 なぜ最速か

HR 払戻 CSV（2015〜2026、`place_1..5_horse/money/pop`）が**取得済み**のため、
新規データ取得ゼロで「複勝市場に正の EV が残っているか」を測定できる。
性能比較で出た「予測1位 単勝ROI +4.1pp（過剰投票回避）」シグナルの検証も兼ねる。

### C-2 実装タスク（順序固定）

1. **`evaluation/place_payout_loader.py`**: race_hr CSV → `race_id × horse_num → 複勝払戻倍率`
   （money は 100 円あたり払戻。race_id 構築は既存 WinOdds ローダーの結合キーと同一規約）
   - TDD: 5頭立て以下（複勝2着まで）・同着・返還の扱いをテストに含める
2. **市場ベースライン拡張**（`evaluation/market_baseline.py`）:
   1番人気の複勝的中率・複勝ROI を `market_baseline.json` に追記
3. **予測1位の複勝ROI 比較**（最重要の一次検証）:
   L1 予測1位に複勝ベット vs 1番人気に複勝ベット（TEST 2025+、fold2 スコア）。
   単勝で +4.1pp だった「過剰投票回避」が複勝でも出るか、100% を超えるかを見る
4. **L2/L3 接続**: `run_backtest_oos.py` の place 解禁
   - `p_place`（既存 Stern λ、fold2 OOS フィット済み）× 複勝払戻倍率で EV
   - VALID=2024 で閾値選択、TEST=2025+ で 1 回判定（既存プロトコル踏襲）
   - 現在の `raise ValueError("place bets disabled...")` は payout loader 接続時のみ解除

### C-3 測定上の注意（正直さの担保）

- HR は**確定払戻**であり、購入時点の複勝オッズではない。まず確定払戻で
  「ROI 上限」を測る（**不可能性テスト**: 確定払戻でも <100% なら事前オッズでは尚更不成立）
- 確定払戻で >100% が出た場合のみ、事前複勝オッズ（O1/O2 系レコード）の取得を
  次タスクとして起票する（JV-Link 接続が必要になるのはこの段階のみ）
- 複勝は的中率が高く分散が小さい → n≥200 ゲートは維持しつつ、Bootstrap 95%CI を併記

### C-4 合格ゲート（Phase 3 複勝版）

| 指標 | 基準 |
|------|------|
| 確定払戻ベース ROI（EV フィルタ後、TEST） | > 100% |
| ベット数 | ≥ 200 |
| キャリブレーション（p_place 10-bin） | 最大誤差 5pp 以内 |
| 不成立時 | 「複勝にも正の EV なし」と記録し §5 へ |

## 4. 優先順位・依存関係・並行チョア

```
Track A（αゲート基盤）─┬─→ Track B（調教特徴量、長期・反復）
                        └─（判定軸として）
Track C（複勝測定）──────→ 独立に即着手可（A に依存しない）

推奨順: C-1〜C-3（数日）と A を並行 → C-4 → B を反復
```

**並行チョア（本計画のスコープ外だが未解決のまま）**:
- L4 当日パイプラインのアーカイブ済み import 残置（`main/main.py`,
  `strategy_pipeline.py`, `view_pipeline.py` ほか）— マスタープラン Phase 5。
  当日運用を再開する前に必須
- 未コミットの v48/v49 実験残骸（`pure_rank/data/02_features/` の JSON 群）の棚卸し

## 5. 全体の撤退・方針転換基準

Track B（5 ファミリ全滅）**かつ** Track C（複勝も EV なし）の場合:

> 「JV-Link 公式データのみからは市場超え（ROI>100%）は不成立」と正式に記録し、
> 以下のいずれかをユーザーに諮る（勝手に着手しない）:
> (a) 外部データ源の導入（パドック評価・馬体重推移の非線形・詳細天候・レース映像由来指標）
> (b) プロジェクト目標の再定義（的中率軸の研究プロジェクトとして継続、ROI 目標を撤回）
> (c) 「損失最小化」運用（ROI 82% は市場ベンチ 77.9% より優位）としての位置づけ

## 6. 実装規約（全 Track 共通）

1. superpowers:test-driven-development 準拠（テスト先行、RED 確認必須）
2. 事前登録 → 実装 → 機械的ゲート判定 → 不合格時ロールバック（従来プロトコル継続）
3. **TEST(2025+) の使用は各候補・各券種 1 回のみ**。閾値・方式選択は 2024 以前で完結
4. 市場情報境界: HC/WC/HR は市場情報ではない（JRA 公式計測・確定結果）。
   ただし HR の `*_pop`（人気）列を L1 特徴量に使うことは禁止（憲法 1 条）
5. 生成物は manifest 付き、レポートは `evaluation/reports/` に集約、gate_summary に反映

## 7. 完了の定義

- Track A: α ゲートがネガ/ポジコントロールを正しく判定し、CLAUDE.md 新プロトコル反映済み
- Track C: 複勝 ROI の正式測定値が gate_summary に PASS/FAIL で記録済み
- Track B: 5 ファミリの γ 判定がすべて記録済み（採用 or 却下 or 撤退）
- いずれの結末でも「何が測定され、何が結論か」が第三者に再現可能な形で残っている
