# α>0 回復計画 — データ拡張仕様書（2026-07-12 planner）

## 背景と目的

2026-07-08〜10 の正式OOS検証で、現行 JV-Link 特徴量セット（v39_course_slim）は
**市場（オッズ）に対する追加情報ゼロ（fold2 完全OOS で α=0）** と実測された。
また 2026-07-12 の調査で以下を確認済み:

- 本番 fusion_params（α≈1.06）は L1 in-sample 期間でフィットされた汚染値であり、
  2025 TEST 実測で **EV バケットと ROI は逆相関**（EV≥1.15 帯 ROI 57〜72% < 平均）。
  EV フィルタ運用は winner's curse を選別する逆効果。
- 回収率 100% 超えの前提は「市場が織り込んでいない情報」の獲得であり、
  モデル・購入法の改良では到達できない。

**本計画の目的**: 市場が織り込みきれていない可能性のある新規データを L1 に追加し、
**fold2 完全OOSプロトコルで α>0 を実測すること**。α>0 が出るまで券種展開・
執行レイヤーへの投資は行わない（エッジゼロのまま組み合わせ市場へ行くと損失が増幅されるため）。

## 成功基準（ゲート）

fold2 完全OOSプロトコル（L1: train≤2022 / ES=2023 / TEST=2024、
`prob_fusion/src/run_fit_oos.py` 準拠）で:

1. **α > 0**（信頼区間が0を含まない）かつ LRT p < 0.05
2. **test logloss（fusion）< test logloss（market）**
3. 参考: モデルTop-1 への flat 単勝 ROI が 1番人気ベースラインを paired bootstrap CI95 で上回る

いずれも TEST（2024）は判定1回のみ使用（Rule 3）。閾値・特徴量の調整は train/valid のみ。

## Phase A: クッション値・含水率・馬場バイアス特徴量（最優先・既存データで着手可）

### データ
`common/data/output/cushion/cushion_all.csv`（12,792行、2018〜2025）
- `moisture_pct`（含水率）: 2018〜 全行あり。芝・ダート別、測定地点別
- `cushion_value`: 2020-09 導入（2021以降は芝行に完備）。**現在完全に未使用**
- 両方とも**当日朝にJRAが発表する値**（レース前確定）→ 時系列リークなし、
  市場情報でもない → L1 特徴量として憲法適合

### 特徴量仕様（市場情報不使用）
| # | 特徴量 | 定義 | リーク対策 |
|---|--------|------|-----------|
| A1 | `cushion_value` | 当日・当該場・芝の値（ダートはNaN） | 当日朝発表値のみ |
| A2 | `moisture_pct` | 当日・当該場・当該サーフェスの含水率 | 同上 |
| A3 | `cushion_diff_track_avg` | 当日値 − 当該場の過去1年平均（当日除く） | shift必須 |
| A4 | `hist_perf_similar_cushion` | 当該馬の過去走のうちクッション値±0.5帯での複勝率 | shift(1) expanding |
| A5 | `hist_perf_similar_moisture` | 同・含水率帯（ダート含む） | shift(1) expanding |
| A6 | `last3f_rank_x_cushion` | 過去走上がり順位平均 × 当日クッション値（軟らかい馬場での末脚適性） | 過去走はshift済み |

- 欠損（2020-09以前のクッション値、障害等）は NaN のまま LightGBM に渡す
- カテゴリではなく連続値として投入。`train_config.json` に feature list 追加

### 馬場バイアス（Phase A2、A1〜A6 の後）
同日・同場・先行レースの「枠番/通過順位別の着順残差」から内外・前後バイアスを日次推定。
- 使用可能なのは**当該レースより発走時刻が前のレースのみ**（同日 shift）
- 当日1〜2Rでは欠損 → NaN 許容
- 実装コストが高いため、A1〜A6 の単体効果測定後に着手判断

### 評価手順
1. implementer: fold2 OOS用 L1再学習（train≤2022 / ES=2023）で A1〜A6 追加版を学習
2. evaluator: Top-1/NDCG/Spearman を v39 fold2-OOS ベースラインと比較（参考指標）
3. **主判定**: run_fit_oos.py で α 再フィット → 上記ゲート

## Phase B: 時系列オッズ収集（取得のみ即時開始・利用は後段）

### 確認済み仕様（docs/JV-Data.md）
- dataspec `0B41`（単複枠 O1）・`0B42`（馬連 O2）: レース毎・複数時間帯の中間オッズ
- O1/O2 の「発表月日時分」フィールドが時系列キー
- **保持期間1年間** → 遡及取得は過去1年分が上限。**取得開始が遅れるほど失われる**
- ワイド・馬単・三連系に時系列提供はない（速報 0B30 系のみ、リアルタイム限定）

### タスク
1. implementer: `common/data/src/` に 0B41/0B42 フェッチャ実装
   - 初回: 過去1年分（2025-07〜）一括取得 → `common/data/output/odds_ts/` に parquet 保存
   - 以後: 週次で差分取得（金土日分）。スキーマ: race_id, horse_num（/組番）, 発表時刻, odds
2. 蓄積のみ。分析・特徴量化は行わない

### 将来の用途（憲法上の位置づけ）
- **L1 特徴量には使わない**（市場情報境界）
- L3/執行レイヤー限定:
  - オッズドリフト分析（前日→直前の変動と結果の関係、スマートマネー検出）
  - 購入時点オッズでの執行シミュレーション（現行バックテストは確定オッズ）
- 検証には最低1年分の蓄積が必要 → 収集を先に始める理由

## Phase C: 追加データ候補（Phase A の結果を見て判断）

## 本番運用への即時勧告（本計画と独立）

- 本番の当日推奨は `mode: loss_min_top1`（flat Top-1）を維持すること
- **EV 表示・EV フィルタ由来の推奨は α 汚染が解消されるまで参考出力からも除外を推奨**
  （EVと実ROIが逆相関している実測があるため、ユーザーの判断を誤導する）
