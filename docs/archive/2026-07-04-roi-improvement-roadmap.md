## 1. 今回の新事実の分析

### 1.1 Top-1改善（29.93%→30.24%、v37→v39）がROIに直結しなかった

| 指標 | v33系（Top-1=30.37%） | v39_course_slim（Top-1=30.24%） | 差分 |
|------|-----|-----|------|
| ワイド 全件 ROI | 80.99% | 80.64% | -0.35pp |
| ワイド EV≥1.0 ROI | 85.72%（n=1,478） | 83.28%（n=1,535） | -2.44pp |
| 馬連 全件 ROI | 78.11% | 79.66% | +1.55pp |
| 馬連 EV≥1.0 ROI | 78.30%（n=1,876） | 79.45%（n=1,959） | +1.15pp |

ワイドは悪化、馬連は改善と方向が一致しない。Top-1の差はわずか0.13pp（v33系はむしろv39より高い）であることを踏まえると、これは「ランキング精度の変化がROIに系統的に効く」という仮説を支持しない。**ランキング精度とROIは別の指標である**という既存知見（メモリ: feedback_ev_strategy.md）が改めて定量的に裏付けられた。

**含意**: Top-1をさらに0.6〜2.8pp（市場ベンチマークの31〜33%まで）引き上げる努力を続けても、ROI改善が自動的についてくる保証はない。ROI改善はランキング精度向上と**切り離した別トラック**として扱う必要がある（これは既存方針どおりだが、今回の数値で再確認された）。

### 1.2 R-7相当の診断は既に一度実施済み — 結果は明確な「否」

`docs/specs/2026-07-01-ev-roi-correlation-diagnosis.md` で設計された条件別EVリフト診断（`simulate_ev.py --diagnose-ev-conditions`）は**O1（単勝・複勝事前オッズ）を使わず、既存のWide/QuinellaOddsのみ**で実装済みであり、2026-07-02に旧モデル（v33系、Top-1=30.37%）で一度実行されている（`pure_rank/data/02_features/ev_results.json` の `best_condition_sweep`）。結果は次の通り:

| 段階 | 結果 |
|------|------|
| VALID(2024) 8次元×36条件のスクリーニング | 「有効」判定は2件のみ: `course_code=1`（ev_lift=+68.5pp）, `track_condition_code=3`（ev_lift=+24.5pp） |
| 上記2条件のTEST(2025+)独立検証 | `course_code=1`: ev_lift_test **-30.5pp**（ROI 164%→50.5%に反転）<br>`track_condition_code=3`: ev_lift_test **-21.4pp**（ROI 115%→59.6%に反転） |
| 複合フィルタ（OR）のTEST ROI | 58.0%（全件平均80.99%を大きく下回る） |

これは仕様書自身が定義する「期待C」（VALIDで有効だがTESTで無効＝過学習の疑い→**不採用**）にそのまま該当する。**条件ベースのEVフィルタ探索は、一度実施され、公式な事前登録プロトコルに従って否定された**という事実がある。

さらに重要な副産物として、**weather_code=3（雨）の同じVALIDスクリーニングでの結果**を確認したところ:

| dimension | value | roi_all (VALID) | roi_ev_filtered (VALID) | ev_lift (VALID) | 判定 |
|-----------|-------|-----------------|--------------------------|------------------|------|
| weather_code | 3 | 54.9% | 37.9% | **-17.1pp** | 無効（有効条件リストに不採録） |

つまり雨条件はVALID(2024)の時点で**むしろ負のリフト**であり、「有効」判定を通過していない。TEST側で観測された ROI=126.8%（n=60、2026-07-01時点）→102.1%（n=67、今回）という推移は、公式スクリーニングの結果と整合的な「サンプルサイズ増加に伴う平均への回帰（＝ノイズ）」として説明できる。

**含意（雨条件の監視継続判断の再検討）**: `docs/2026-06-30〜07-02` の各ドキュメントは雨条件を「唯一の生存候補」「監視継続」と位置づけていたが、これは2026-07-01時点（VALIDスクリーニング実施前）の暫定判断である。2026-07-02に実施済みの公式スクリーニングの結果（VALID ev_lift=-17.1pp）と、今回のn増加でのROI退行を合わせると、**雨条件を「有望な監視対象」として扱う根拠は消滅している**。以後は「後出しで見つかった条件で、事前登録プロトコルに不合格」として正式にクローズし、新規の追加検証コストをかけない。

### 1.3 両事実が指し示す共通の結論

Top-1改善もROIに効かず、条件フィルタ探索もVALID→TESTで再現しない。これは「周辺的な工夫（ランキングの微改善・ベット条件の絞り込み）では市場に対する安定的なエッジを作れない」ことを強く示唆する。`ev_diagnosis_v2.json` が示す**Harville確率の系統的な過大推定（全binでpredicted > actual、最大16.6pp）**が依然として未対処であり、これがEVフィルタが機能しない根本原因である可能性が最も高い。

### 意思決定ルール（R-1の結果による分岐）

```
R-1（Stern化）実施 → 2025+テストで検証
  ├─ 成立（キャリブレーション誤差縮小 AND ROI改善）
  │     → R-3（帯別フィルタ、Stern確率ベースで再設計）へ進む
  │     → O1到着後にR-2・full R-7で確率精度をさらに検証
  │
  └─ 不成立（λ≈1 に収束 or 誤差縮小してもROI改善せず）
        → EVフィルタ経路（R-2, R-3, R-5）は現行の純粋能力スコアでは打ち切り

        → ランキング精度向上（D-1/D-2系）はROIと無関係に継続
```

この分岐は `docs/2026-07-02-system-issues-and-improvements.md` R-7の「意思決定ルール」と同型だが、判定対象をR-1（確率モデル自体の改善）に変更している。理由は1.2/1.3で示した通り、条件フィルタ探索（旧R-7の主眼）は既に一度失敗しており、次に潰すべき仮説はP_wide自体の精度だからである。

### ステップ2（本命・中コスト）: R-1 Harville→Stern化

**担当**: implementer（実装）→ evaluator（検証）

**実装対象**: `pure_rank/src/predict.py`

1. Stern型確率変換関数を追加（Harvilleは削除せず併存させ、比較可能にする）:
   ```python
   def stern_second_prob(p: np.ndarray, winner_idx: int, lam: float) -> np.ndarray:
       q = p ** lam
       q[winner_idx] = 0.0
       return q / q.sum()
   ```
   3着確率も同様の構造で `lam3` を用いて拡張する（2着が決まった後の3着争いに対して独立にλ3を推定）。

2. λ2・λ3のフィット手順:
   - データ: TRAIN+VALID（〜2024-12-31）のみ。TEST（2025+）は使わない
   - 目的関数: 実際の2着・3着馬に対するStern型確率のlog-likelihood最大化（グリッドサーチ、例: λ∈[0.5, 1.0]を0.02刻み）
   - 既存の `T_opt` フィット（`_search_temperature`）と同じ「粗探索→細探索」パターンを踏襲する
   - 結果を `train_config.json` の `plackett_luce` セクションに `lam2_opt`, `lam3_opt` として追加保存する（`T_opt` と同じ管理方式）

3. `simulate_ev.py` 側でHarville版とStern版のP_wide/P_quinを両方計算できるようにし、`--prob-method {harville,stern}` のようなフラグで切り替え可能にする（比較評価のため）

**検証基準（2025+テストで一度きり）**:
- (a) キャリブレーション誤差: `ev_diagnosis_v2.json` の最大誤差16.6pp、`ev_results_v39_course_slim.json` の `calibration.max_abs_error`=0.1898（18.98pp）を下回るか

- **失敗基準**: λ2・λ3が1.0近辺（0.95〜1.05）に収束する場合（＝Harvilleで十分だった）、または誤差は縮小してもROIが改善しない場合は「キャリブレーション精度≠EV有効性」の既知パターンとして撤退し、第2節の意思決定ルールに従ってR-6へ移行する

**リーク停止閾値**: 本施策はベッティングレイヤーのみでランキングモデルに影響しないため、Top-1>40%等のリーク停止閾値の対象外。ただし念のためλフィット後もランキング精度（Top-1/NDCG/Spearman）が不変であることを確認する（predict.pyはモデル出力スコアを変換するだけで、学習済みモデル自体は変更しないため理論上不変のはず）。

## 5. リスク・注意点

1. **Rule 3（後出しじゃんけん禁止）**: λ2・λ3、将来のR-3帯別δは必ずTRAIN+VALID（〜2024）のみで決定する。TEST（2025+）データを見てパラメータや条件を調整することは、過去のキャリブレーション失敗（Platt/Isotonic/ROI-T全滅）と同じ轍を踏む。R-1の検証は「2025+で一度だけ判定」を厳守し、不合格なら別の年次データが蓄積するまで再挑戦しない。

2. **キャリブレーション失敗の再発防止**: 過去3手法（Platt/Isotonic/ROI-T）の共通失敗要因は「2024年の誤差曲線を丸ごと写像する」高自由度の手法だったこと。Stern型はパラメータが実質1〜2個（λ2, λ3）のみで構造化されているため過学習耐性が高いと期待されるが、これは仮説であり検証が必須。**λ≈1に収束した場合は「Sternでも直せない系統誤差」と解釈し、確率モデルの根本変更（R-6统合）に切り替える**。中間的な妥協（λをさらに複雑化する等）は避ける。

3. **未評価のまま放置されている手当て**: `train_config.json` の `calibration.method = "bracket_isotonic"`（`fitted: true`）は、`ev_results_v39_course_slim.json` 等で `calibration_comparison: null` のままTESTでの比較評価が実施されていない。これは事実上「4つ目のキャリブレーション手法」であり、過去3手法と同じ失敗パターンをたどる可能性が高い。本ロードマップの主対象ではないが、放置すると「評価されていない設定が本番configに残る」状態になるため、R-1完了後の整理タイミングでrefactorerに棚卸しを依頼することを推奨する（削除するか、TEST比較して正式に不採用記録を残すか）。

4. **雨条件の再燃防止**: 1.2節でクローズと結論づけたが、n=67という小サンプルでの「まだ100%未満」という状態は今後もROI>100%に見える瞬間が発生しうる（ボラティリティによる一時的な好走）。VALIDスクリーニングで既に否定された条件であることをドキュメントに残し、単発の良好な数値だけで再度「監視継続」に格上げしないことを徹底する。

5. **200レース未満での有意判断の禁止**: R-1検証・R-7-lite再実行のいずれも、条件別のサブグループ集計を伴う場合は200レース未満のセグメントで「有意な改善」と主張しない（禁止事項6）。
