## 2. 候補評価

### 候補1: 洋芝（函館・札幌）区別 × 父系統パワー・持続力適性

**文書の主張**: 函館・札幌の洋芝はクッション性が高くパワー・持続力を要求し、
本州の野芝と質的に異なる。ステイゴールド系・ハーツクライ系等のパワー型父系統が
洋芝で相対的に浮上する。

**既存資産との重複評価**:
`course_code ∈ {1, 2}`（札幌・函館）は現行カテゴリ特徴量に完全に内包されている。
さらに D-1 の教訓（`docs/specs/2026-07-03-d1-course-features-design.md` セクション2.1）
により、**コースの静的部分集合フラグ単体は木にとって新情報を持たず、
実際に `course_is_small`（札幌・函館・福島・小倉を含む上位集合）は split=0 で
不採用となった実績がある**。洋芝2場フラグを単体で追加しても同じ失敗を繰り返す
可能性が高い。

文書の主張の本質は「父系統適性 × 特定馬場条件」という**交互作用**であり、
これは候補2（`hist_sire_track_condition_win_rate_ts`）が測定しようとしている
軸と機構的に同一である（洋芝の「クッション性・パワー要求」は track_condition_code
が直接表す「重い/タフな馬場への耐性」の特殊ケースと解釈できる）。
洋芝2場限定の交互作用（例: `hist_sire_surface_win_rate_ts × is_hokkaido_turf`）を
独立候補として追加すると、対象サンプルが札幌・函館の芝レースのみに絞られ
（学習期間で概算 全芝レースの1割未満）、候補2より情報量で劣る派生形になる。

**判断: 実装しない（今回は見送り、条件付き将来検討）**

- 実装コスト: 低（is_hokkaido_turf は course_code ∈ {1,2} & surface_code=1 の静的フラグ）
- 重複リスク: 高（候補2の下位集合。D-1の静的フラグ単体失敗パターンと同型）
- **フォローアップ条件**: 候補2（`hist_sire_track_condition_win_rate_ts`）を実装・評価した後、
  `analyze_errors.py` で「函館・小倉開催時期（course_code ∈ {1,2} かつ月次で開催期間内）の
  Top-1」を副次確認する。候補2採用後もなお函館・札幌が構造的弱点として残る場合に限り、
  `hist_sire_surface_win_rate_ts × is_hokkaido_turf` の交互作用を単独実験として再検討する
  （相関ゲート必須、想定 r は候補2の同型列と同程度〜やや高いと見込む）。

### 候補2: 父×馬場状態別勝率（`hist_sire_track_condition_win_rate_ts`）

**文書の主張**: 梅雨時のタフな馬場・洋芝への耐性は父系統によって差があり、
特に道悪適性はパワー型血統で相対的に有利。

**既存資産との重複評価**: Phase B2（`docs/specs/2026-07-02-phase-b2-strategy-design.md`
候補B）で既に詳細設計済み。`hist_sire_surface_win_rate_ts`（芝/ダートの2値）と
`track_condition_code`（良/稍重/重/不良の4値）は直交する次元であり、
`_build_sire_features` の既存パターン（日次集計→cumsum→shift(1)、`MIN_SIRE_RACES=30`）
をそのまま流用できる。

**留意点（正直な評価）**: Phase B2 設計時点のトリガー条件「track_condition別格差5pp以上」
は不成立だった。さらに直近のエラー分析（v33時点、`project_phase_progress.md` 記載）では
「**道悪はむしろ強い**」（稍重 Top-1=33.21%、全体比+3.1pp）という逆方向の結果が出ている。
これは「道悪で父系統適性が効いていないから弱い」という文書の前提とは異なり、
むしろ「既存特徴量群（クラス・距離・展開系）が道悪でも十分機能している」ことを示唆する。
したがって候補2は「弱点解消」ではなく「独立軸の追加による一般的な精度底上げ」という
位置づけで評価すべきであり、過度な期待（大幅改善）は禁物である。

ただし文書は「道悪一般」ではなく「梅雨時の消耗戦・洋芝のパワー要求」という、
より狭い条件下の血統的差別化を主張しており、これは track_condition_code という
既存の粗いカテゴリでは捉えきれない可能性がある独立仮説として、依然として
試験する価値はある（実装コストが極めて低いため）。

**判断: 実装推奨（優先度1）**

- 実装コスト: 低（既存パターンのコピー、`_build_sire_features` に追加するのみ）
- 重複リスク: 中〜低（surface と track_condition は直交するが、相関ゲートで実測必須。
  想定 r は `hist_sire_surface_win_rate_ts` との間で r=0.3〜0.5 程度、
  `hist_sire_dist_win_rate_ts` とはほぼ無相関と見込む）
- リーク防止: 既存 `hist_sire_surface_win_rate_ts` と完全同型（日次集計→shift(1)）。
  `track_condition_code=0`（不明）は NaN 扱いで問題ない
- 期待改善: 0〜0.2pp（控えめ。上記の理由で過度な期待をしない）
- バージョン: **v43_sire_tc**

### 候補3: ハンデ戦フラグ（`weight_type` 由来）

**文書の主張**: ハンデ戦は斤量格差を人為的に均す制度であり、出走馬間の実力差を
縮小させるため、上位人気馬の信頼度が下がり2着・3着に人気薄が絡みやすい。

**除外理由の確認（最重要）**: `common.py` の `FORBIDDEN_COLS` で `weight_type` は
「RA/SEのメタ列（特徴量として不要）」という**整理上の除外**として分類されており、
市場情報だからではない。`weight_type`（重量種別コード。JV-Link RA レコード、
オフセット622、1桁）は定量/別定/馬齢/ハンデキャップ等の**競走条件コード**であり、
出走馬確定時にJRAが公表するレース番組情報である。オッズ・人気とは無関係の
制度情報であることを確認した。

**リーク安全性**: ハンデキャップの「値」（各馬の負担重量）自体はハンデキャッパーが
馬の近走成績を見て決定するため能力の代理指標になりうるが、ここで提案するのは
「このレースがハンデ戦か否か」という**レース単位の制度フラグ**であり、当該馬の
負担重量の絶対値ではない。フラグ自体はレース施行が公示された時点（出走前）で
確定しており、時系列リークはない。

**既存資産との重複評価**: `class_code` / `race_level` / `grade_code` 等の
クラス系特徴量と一定の相関がある可能性がある（ハンデ戦は特定クラス帯に偏在する
制度慣行があるため）。ただし「クラスが同じでもハンデ戦か定量戦かでフィールドの
実力分散が変わる」という文書の主張は、クラス情報だけでは表現できない独立軸である。

**設計**:
- `is_handicap_race`: `weight_type` のうちハンデキャップに対応するコード値が1、
  それ以外0の二値フラグ（int8）。**コード値は実装時に `weight_type` の value_counts
  と JV-Link 仕様書を突き合わせて確定する**（D-1 の `track_code` value_counts 検証と
  同じ流儀。目視憶測でコード番号を決め打ちしない）
- カテゴリ特徴量には追加しない（二値のため数値列で十分）

**判断: 実装推奨（優先度2）**

- 実装コスト: 低（静的フラグ、集計不要）
- 重複リスク: 中（class_code / race_level と相関ゲートで確認必須。想定 r=0.2〜0.4程度）
- リーク防止: 該当なし（レース前確定の制度情報）
- 期待改善: 0〜0.15pp（「上位人気の信頼度低下」という文書の主張はROI・波乱度の話であり、
  純粋ランキング精度＝Top-1的中率にどう転写されるかは不明瞭。過度な期待をしない）
- バージョン: **v44_handicap**

### 候補4: 輸送距離・滞在競馬フラグ（`region_code` 由来）

**文書の主張**: 長距離輸送ストレスは能力発揮を妨げる。北海道シリーズでの滞在競馬
（現地滞在によって輸送ストレスを回避する調教方針）はこの逆の効果を持つ。

**既存資産との重複評価**: `region_code`（東西所属コード。SEレコード、オフセット85、
1桁。馬の所属トレセンが美浦[東]か栗東[西]かを表す）は現行 `FORBIDDEN_COLS` に
「メタ列」として分類されているが、市場情報ではなく馬の登録情報である。
特徴量として未使用であることを確認した。

**設計上の制約（正直な評価）**: `region_code` 単体では「東/西所属」の2値（+地方所属等の
例外値）しか分からず、実際の輸送距離・滞在競馬の有無までは判別できない。
文書が指摘する「滞在競馬による輸送ストレス排除」は、JV-Linkの標準データからは
直接検出できない（滞在の有無はレース前提でトレセンからの直接輸送か、現地
滞在厩舎からの輸送かを区別する情報がない）。

したがって実装可能な範囲は次の粗い近似にとどまる:

- `transport_category`: `region_code`（所属）× `course_code`（開催地）の組み合わせから
  「近郊開催（同一地域ブロック内）」「中距離移動」「長距離移動（例: 西所属馬の
  札幌・函館遠征、東所属馬の九州圏遠征）」の3カテゴリに粗く区分した静的カテゴリ変数
- マッピングは学習期間データではなく、**地理的事実**（トレセン所在地と競馬場の
  実距離）のみに基づいて事前に固定する（後出し調整を避けるため）

**リーク安全性**: `region_code` は馬の恒常的な所属情報（レース結果に依存しない）、
`course_code` は開催が決定した時点で確定（レース前情報）。両者の組み合わせに
時系列リークはない。

**判断: 実装推奨（優先度3、期待値は控えめ）**

- 実装コスト: 中（地理マッピングテーブルの新規作成が必要。COURSE_GEOMETRY と同様に
  モジュール定数として実装、チューニング対象ではない）
- 重複リスク: 低〜中（`course_code` カテゴリ・`course_is_small` 系との相関を確認。
  ただし `transport_category` はカテゴリ変数として扱う場合は Pearson 相関ゲートの
  対象外になるため、実装時に数値順序変数にするかカテゴリにするかを決定し、
  カテゴリにする場合は `train_config.json` の `categorical_feature` リストへの追加要否を
  implementer が判断する）
- 期待改善: 0〜0.1pp（JRAの輸送は航空便・専用車両で標準化されており、文書が想定する
  「輸送ストレス」の効果量自体がJRAでは地方競馬ほど大きくない可能性がある。
  函館・札幌・小倉開催での効果に限定的に期待する）
- バージョン: **v45_transport**

### 候補5（将来検討・今回は設計のみ）: 当日気温・暑熱指数

**文書の主張**: 熱ストレスによる能力低下が夏競馬の不確実性の一因。

**判断: 今回は実装しない（長期検討事項として登録するのみ）**

- JV-Linkの標準データ（RA/SE/HC/PED/TM）には気温・暑熱指数が含まれておらず、
  気象庁等の外部データソースの取得が必要
- 外部データ取得は本仕様書のスコープ外（データパイプライン新設が必要、工数大）
- 実装する場合の設計要点（将来の参考）:
  - 気象庁の競馬場所在地に最も近い観測地点の日別最高気温・WBGT相当値を取得
  - レース前確定情報であることは自明（気温はレース結果と独立）だが、
    **外部データの結合キー（race_date × course_code → 観測地点）の正確性**が
    リーク防止上の唯一の論点になる
  - 取得スクリプトは `common/data/src/` に新設し、JV-Linkとは別系統のデータ取得
    タスクとして切り出す（本仕様書は設計のみで実装タスクを起票しない）

## 3. 実験プロトコル（1変更1実験）

```
実験1: v43_sire_tc   （hist_sire_track_condition_win_rate_ts、1列）
   ↓ evaluator 合否判定
実験2: v44_handicap  （is_handicap_race、1列）      ← ベース: 実験1の採否に応じて更新
   ↓ evaluator 合否判定
実験3: v45_transport （transport_category、1列）    ← ベース: その時点の採用ベースライン
   ↓ evaluator 合否判定
（条件付き）実験4: 候補2の結果と函館・札幌の副次分析次第で
              hist_sire_surface_win_rate_ts × is_hokkaido_turf を再検討
```

- 各候補は独立仮説であり `NEW_FEATURE_COLS_BY_VERSION` に単独登録して個別に相関ゲートを通す
  （D-1実験1のような単一仮説バンドルではないため、3実験を束ねない）
- 各実験は 5シード×3フォールドのフルアンサンブル（15モデル）で評価する。
  シード・フォールド・label_gain・num_leaves 等の学習設定は一切変更しない
  （`train_config.json` の `features_version` のみ変更）
- 実験1が合格しても不合格でも、実験2は「その時点の採用ベースライン」
  （合格なら v43_sire_tc、不合格なら v39_course_slim）に対して追加する

### 相関ゲートへの接続

既存の `_run_correlation_gate()`（`pure_rank/src/create_features.py` セクション8.5）を
そのまま使う。各バージョンについて `NEW_FEATURE_COLS_BY_VERSION` に追記する:

```python
NEW_FEATURE_COLS_BY_VERSION: dict[str, list[str]] = {
    ...
    "v43_sire_tc":    ["hist_sire_track_condition_win_rate_ts"],
    "v44_handicap":   ["is_handicap_race"],
    "v45_transport":  ["transport_category"],  # カテゴリ扱いにする場合は数値化した順序版で相関ゲートにかける
}
```

`CORR_GATE_THRESHOLD = 0.7` は変更しない。違反した候補は学習に進まず、その候補は
不採用として記録し次候補に進む（Phase A・D-1と同じ運用）。

## 4. 評価基準（全候補共通）

| 指標 | 合格 | 不合格 |
|------|------|--------|
| Top-1 | > 30.24%（v39_course_slim 超え） | <= 30.24% |
| NDCG@3 | >= 0.5329（v39 比 -0.003 以内） | < 0.5329 |
| Spearman | >= 0.4998（v39 比 -0.005 以内） | < 0.4998 |

3条件すべて満たして合格。Top-1のみ改善してNDCG@3・Spearmanが許容幅を超えて
悪化する場合は不合格とする（ランキング全体の質を犠牲にしない、D-1と同じ運用）。

**リーク停止閾値**: Top-1 > 40% または Spearman > 0.6 → 即座に実装停止し evaluator へ報告
（合格ではなく危険信号として扱う）。

### 副次評価（合否には使わない。方向性の参考のみ）

各候補が合格した場合、`analyze_errors.py` で以下を確認する（多重検定的な後付け
選択を避けるため、これらはゲートではなく記録のみ）:

| 条件 | 現行値（v39_course_slim） | 期待方向（候補との対応） |
|------|--------------------------|------------------------|
| 稍重・重・不良（track_condition） | 稍重33.21%（+3.1pp） | 候補2: 悪化していないこと |
| 夏季開催（race_date月が6〜9月） | 未測定（本仕様書で新設） | 候補1系: 改善方向が望ましいが必須ではない |

1. バックアップ確認: `features_v39_course_slim.parquet` の存在確認、
   `pure_rank/models/` の15モデルを退避

### 実装上の注意

- 候補2の実装位置: `_build_sire_features` 内、既存の `hist_sire_surface_win_rate_ts` を
  生成するブロック（`create_features.py` 616〜637行付近）の直後に追加。
  `MIN_SIRE_RACES = 30` の閾値は共通利用する
- 候補3の `is_handicap_race` は int8 の二値列とし categorical には追加しない
- 候補4の `TRANSPORT_MAP` は地理的事実（トレセン所在地と競馬場所在地の関係）のみに
  基づき固定する。学習期間データを見てカテゴリ境界を調整しない
  （後出し調整の禁止に抵触するため）
- 実験間で `features_version` 以外の設定を変えない。複数候補の同時追加は行わない
  （1変更1実験の原則）
- 候補1（洋芝×血統の交互作用）は本仕様書の実験プロトコルに含めない。
  候補2の結果確定後、セクション2の「フォローアップ条件」に従って
  別途 planner が実装要否を判断する
